# 项目初始化操作手册

这份手册专门解决一个问题：

- 人应该怎么使用 `_agent_blueprint` 这套模板，去启动一个全新的项目

它不是技术细节大全，而是一份“按步骤照做”的操作说明。

---

## 1. 这份手册适合谁

适合下面这些场景：

- 你自己隔一段时间后重新起新项目
- 你要把这套模板交给同事或朋友使用
- 你希望别人不看聊天记录，只看文档也能完成初始化
- 你希望 Agent 参与初始化，但不想靠临场口述解释需求

---

## 2. 使用这套模板前要先理解什么

这套模板的核心不是“直接生成成品项目”，而是先建立一份让 Agent 和人都能读懂的项目契约。

如果你第一次打开 `_agent_blueprint`，不确定先看哪份文档，建议先看：

```text
doc/01-onboarding/使用总览导航.md
```

它会先告诉你不同场景下的推荐阅读顺序，再回到这份操作手册继续执行。

它解决的是：

- 项目需求怎么表达
- Agent 先读什么
- 技术选型怎么收敛
- 环境怎么检查
- 初始化怎么统一执行

一句话理解：

- 先写清楚
- 再让 Agent 理解
- 再让 Agent 动手

补充一个关键原则：

- 根目录文档是默认规则入口
- `templates/` 是可选成品样例，不是每次都必须读取的默认上下文
- 在你还没决定采用哪个模板前，不要一上来就让 Agent 深读所有模板

---

## 3. 模板目录中每个文件是干什么的

你会看到这些核心文件：

- `AGENTS.md`
  - 定义 Agent 应该如何思考、读取、推荐、补全配置

- `README.md`
  - 定义这套模板的目标、默认规则、推荐提示词和使用方式

- `doc/02-workflow/任务执行与交付验收准则.md`
  - 定义 Agent 做任务时的执行边界、交付标准和完成定义
  - 当你希望 Agent 少走偏路、少返工、提高交付质量时，这份文档很关键

- `doc/02-workflow/任务输入模板.md`
  - 定义用户如何更清楚地给 Agent 下任务
  - 当你不想来回补充说明、希望 Agent 更快进入正确方向时，这份文档很有用

- `doc/task-playbooks/`
  - 定义常见具体任务的标准执行顺序
  - 当你已经知道当前任务属于哪一类时，可以让 Agent 直接按对应手册执行

- `doc/02-workflow/任务结果输出模板.md`
  - 定义 Agent 在任务完成后应如何汇报结果
  - 当你希望交付说明更稳定、更易读、更方便复查时，这份文档很有用

- `project-spec.example.yaml`
  - 项目需求模板
  - 你需要复制成 `project-spec.yaml` 再填写

- `.env.example`
  - 环境变量模板
  - 真实项目中通常复制成 `.env`

- `compose.yaml`
  - 本地基础设施模板
  - 默认包含 `PostgreSQL` 和 `Redis`

- `Makefile`
  - 统一命令入口

- `scripts/check-env.sh`
  - 检查本机环境是否具备运行条件
  - 如果发现缺失工具，会先用中文提示，再询问是否自动安装

- `scripts/bootstrap.sh`
  - 推荐新手优先使用的主入口
  - 会先检查环境，再初始化项目和基础设施

- `scripts/dev.sh`
  - 开发运行入口

- `scripts/test.sh`
  - 测试入口

## 4. 新项目的标准启动顺序

推荐你每次都按下面顺序来。

### 第 1 步：复制模板

在你的项目根目录中复制模板。

示例：

```bash
cp -r _agent_blueprint my-new-project
cd my-new-project
```

说明：

- `_agent_blueprint` 是模板目录
- `my-new-project` 只是示例项目名
- 你可以改成任何符合自己习惯的名字

如果你不想直接复制，也可以手动新建项目目录，再把模板文件复制进去。

如果你当前只是想搭建“规则体系”而不是直接起业务项目，可以先只使用根目录文档、脚本和 `project-spec.yaml`，暂时不要使用 `templates/` 下的成品样例。

---

## 5. 第 2 步：生成你的项目需求文件

执行：

```bash
cp project-spec.example.yaml project-spec.yaml
```

然后编辑：

```bash
project-spec.yaml
```

你至少要改这几部分：

- `project.name`
- `project.type`
- `requirements.summary`
- `requirements.core_features`
- `constraints.priorities`
- `preferences`

不要一开始就追求写很复杂。

先把最关键的内容写清楚：

- 这个项目要做什么
- 谁在用
- 核心功能是什么
- 优先简单还是优先性能
- 偏向 Python 还是 Node
- 要不要数据库
- 要不要缓存

---

## 6. 第 3 步：复制环境变量模板

执行：

```bash
cp .env.example .env
```

说明：

- `.env.example` 是示例配置
- `.env` 是你本地真实配置
- 如果后面接 AI 服务，要把 API Key 填到 `.env`

注意：

- `.env` 不应该提交到 Git

---

## 7. 第 4 步：先不要急着写代码

这一步很关键。

在这套工作流里，正确顺序不是：

- 先写代码
- 再补配置

而是：

- 先让 Agent 读规则和需求
- 先确认技术选型
- 再补项目骨架
- 最后才开始业务代码

这样可以减少后面返工。

---

## 8. 第 5 步：给 Agent 下第一条指令

这是推荐的第一条提示词：

```text
先阅读 AGENTS.md、project-spec.yaml、.env.example、compose.yaml 和 scripts 目录。
基于项目需求、约束和偏好，推荐最适合的技术栈。
推荐时优先考虑简单、稳定、便于单人维护和后续扩展。
如果有关键冲突或存在多种差异很大的方案，再向我提问。
```

这条提示词的作用是：

- 先让 Agent 理解项目
- 不让它一上来就盲目生成代码
- 先做技术选型，再做骨架设计
- 默认不把 `templates/` 中的成品代码带入当前上下文

如果你希望 Agent 从一开始就按更高标准执行任务，可以把这份文档也加入首轮阅读范围：

```text
先阅读 AGENTS.md、project-spec.yaml、README.md、doc/02-workflow/任务执行与交付验收准则.md、.env.example、compose.yaml 和 scripts 目录。
基于项目需求、约束和偏好，推荐最适合的技术栈，并按交付准则说明你的判断依据、执行顺序和后续验证方式。
```

如果当前任务已经很具体，比如“新增接口”或“接入第三方 API”，建议再加一句：

```text
如果 doc/task-playbooks/ 中存在与当前任务匹配的操作手册，请按对应手册执行。
```

如果你还希望 Agent 在最终回复时保持统一结构，可以再补一句：

```text
最终结果请参考 doc/02-workflow/任务结果输出模板.md 组织说明，明确写出已完成内容、修改位置、验证情况、风险边界和下一步建议。
```

如果你发现自己经常不知道怎么开口描述任务，可以先参考：

```text
doc/02-workflow/任务输入模板.md
```

先把目标、范围、限制和完成标准写清楚，再交给 Agent。

---

## 9. 第 6 步：让 Agent 生成项目骨架

当 Agent 给出推荐栈后，再下第二条提示词：

```text
先阅读 AGENTS.md 和 project-spec.yaml。
根据需求补全项目骨架、依赖文件、README、.gitignore、.env.example、测试入口和开发脚本。
优先使用统一脚本入口，不要假设机器上已经有全局环境。
完成后告诉我你选择了什么技术栈、为什么这样选、下一步如何运行。
```

这一步完成后，Agent 通常应该补出：

- `pyproject.toml` 或 `package.json`
- `src/`
- `tests/`
- 项目入口文件
- 更具体的 `README.md`
- 更准确的 `compose.yaml`

---

## 10. 第 7 步：优先执行 bootstrap

对大多数第一次使用的人，最简单的方式不是分两步，而是直接执行：

```bash
bash scripts/bootstrap.sh
```

或者：

```bash
make bootstrap
```

这个脚本现在会按下面顺序自动处理：

- 先检查你的机器环境
- 用中文告诉你已经装了什么、还缺什么
- 如果缺少工具，会先询问你是否同意自动安装
- 只有你确认后，才会继续安装缺失项
- 环境准备好后，再继续项目初始化

这一步更适合技术小白，因为只需要记住一个主入口。

---

## 11. 第 8 步：如果需要，再单独运行环境检查

在项目目录中执行：

```bash
bash scripts/check-env.sh
```

或者：

```bash
make check
```

这一步的意义：

- 检查你的机器环境是不是满足项目要求
- 发现少了哪些工具
- 在真正初始化前尽早发现问题
- 如果工具缺失，先用中文说明，再让你确认是否自动安装

什么时候适合单独运行这一步：

- 你只想先看看电脑环境缺了什么
- 你暂时不想立刻初始化项目
- 你想先做环境体检，再决定下一步

---

## 12. 第 9 步：bootstrap 一般会做什么

当 `scripts/bootstrap.sh` 完成环境检查后，一般会继续做这些事：

- 复制 `.env`
- 创建 Python 虚拟环境
- 安装依赖
- 安装 Node 依赖
- 启动 Docker 基础设施

注意：

- 具体行为取决于项目实际生成了哪些文件
- 比如没有 `pyproject.toml`，就不会走 Python 初始化

---

## 13. 第 10 步：启动开发模式

执行：

```bash
bash scripts/dev.sh
```

或者：

```bash
make dev
```

如果脚本只输出提示，不代表失败。

因为模板本身是通用协议，不一定知道你最终项目的启动命令。

这时你应让 Agent 继续补全：

```text
请根据当前项目的实际技术栈，完善 scripts/dev.sh、scripts/test.sh 和 README.md，
让项目可以通过统一入口启动和测试。
```

---

## 14. 第 11 步：跑测试

执行：

```bash
bash scripts/test.sh
```

或者：

```bash
make test
```

如果还没有测试体系，脚本会提示你继续补。

推荐给 Agent 的提示词：

```text
请根据当前项目技术栈补全最小测试体系，并完善 scripts/test.sh。
优先选择简单、可维护的测试工具，不要过度设计。
```

---

## 15. 第 12 步：初始化 Git

如果项目还是新目录，建议尽早做：

```bash
git init
git add .
git commit -m "chore: init project"
```

然后尽早关联远程仓库并推送。

这样做的好处：

- 防止本地修改丢失
- 环境和项目骨架可以快速恢复

---

## 16. 推荐的完整对话顺序

下面是一套最推荐的 Agent 协作顺序。

### 第一轮：理解需求和选型

```text
先阅读 AGENTS.md、project-spec.yaml、.env.example、compose.yaml 和 scripts 目录。
基于项目需求、约束和偏好，推荐最适合的技术栈。
推荐时优先考虑简单、稳定、便于单人维护和后续扩展。
如果有关键冲突或存在多种差异很大的方案，再向我提问。
```

### 第二轮：生成骨架

```text
先阅读 AGENTS.md 和 project-spec.yaml。
根据需求补全项目骨架、依赖文件、README、.gitignore、.env.example、测试入口和开发脚本。
优先使用统一脚本入口，不要假设机器上已经有全局环境。
完成后告诉我你选择了什么技术栈、为什么这样选、下一步如何运行。
```

### 第三轮：检查环境并初始化

```text
先阅读 AGENTS.md、README.md、project-spec.yaml 和 scripts 目录。
优先执行 scripts/bootstrap.sh 作为统一入口。
如果发现缺少工具，请先用中文说明缺什么，并在安装前征求我的确认。
如果我同意，再自动安装缺失项并继续初始化。
如果当前只适合做环境体检，再单独执行 scripts/check-env.sh。
```

### 第四轮：补齐开发和测试入口

```text
请根据当前项目的实际技术栈，完善 scripts/dev.sh、scripts/test.sh 和 README.md，
让项目可以通过统一入口启动、测试和日常开发。
```

## 17. 什么时候需要你自己做决定

虽然 Agent 可以自动推荐，但下面几类问题最好由你拍板：

- 是否前后端分离
- 后端用 Python 还是 Node
- 数据库是否必须是 PostgreSQL
- 是否需要 Redis
- 部署目标是 VPS、云函数还是本地应用
- 是否要优先开发效率，还是优先长期扩展

如果你不确定，也可以先让 Agent 给建议，再由你确认。

## 18. 分享给别人时怎么说

你可以直接把这段话给对方：

```text
这是一个通用项目初始化模板，不绑定某台电脑、某个固定路径或某种唯一技术栈。
你需要先填写 project-spec.yaml，再让 Agent 读取 AGENTS.md 和相关脚本。
默认优先执行 scripts/bootstrap.sh，它会先检查环境，再用中文提示缺失项，并在安装前征求确认。
Agent 会先推荐技术栈，再补全项目骨架和初始化脚本。
模板中的目录名、项目名、数据库名和路径都只是示例，可以按你的实际环境调整。
```

## 19. 最常见误区

### 误区 1：复制模板后立刻开始写业务代码

更好的做法：

- 先写 `project-spec.yaml`
- 先让 Agent 选型并生成骨架

### 误区 2：把示例路径当成固定路径

更好的做法：

- 把路径、命名、容器名都看作默认值
- 按自己的环境调整

### 误区 3：让 Agent 直接“随便搭一个项目”

更好的做法：

- 先让 Agent 阅读规则和需求
- 再让它生成项目结构

### 误区 4：只让 Agent 生成代码，不补脚本和 README

更好的做法：

- 统一把 `README.md`、`scripts`、`.env.example` 一起补齐

### 误区 5：还没选模板，就把 `templates/` 全部交给 Agent

更好的做法：

- 先让 Agent 只基于 `AGENTS.md`、`project-spec.yaml`、`README.md` 和脚本工作
- 只有确定要采用某个成品模板时，再读取对应模板目录

### 误区 6：一定要先手动跑 `check-env.sh`，再跑 `bootstrap.sh`

更好的做法：

- 对大多数人来说，直接运行 `bootstrap.sh` 就够了
- `bootstrap.sh` 会先检查环境，再继续初始化
- `check-env.sh` 更适合单独做环境体检时使用

## 20. 最短执行清单

如果你已经熟悉流程，可以只按这份最短版走。

### 人的操作

```bash
cp -r _agent_blueprint my-new-project
cd my-new-project
cp project-spec.example.yaml project-spec.yaml
cp .env.example .env
```

### 先编辑

- `project-spec.yaml`

### 再给 Agent 的第一条提示词

```text
先阅读 AGENTS.md、project-spec.yaml、.env.example、compose.yaml 和 scripts 目录。
基于项目需求、约束和偏好，推荐最适合的技术栈。
```

### 再执行

```bash
make bootstrap
make dev
make test
```

如果你只是想先看看环境是否缺工具，再单独执行：

```bash
make check
```

## 21. 一句话记忆

- 人先写需求
- Agent 先读规则
- 再做选型
- 再生成骨架
- 再运行 `bootstrap.sh`
- 让脚本自动检查环境并在安装前先征求确认
- 最后统一用脚本开发和测试
