# 任务执行与交付验收准则

这份文档用于回答四个问题：

- Agent 接到任务后应该先做什么
- Agent 在什么情况下可以直接做，什么情况下必须先确认
- 一次高质量交付至少应包含什么
- 什么叫“任务真的完成了”

它既适用于 Agent，也适用于人类协作者。

## 1. 总原则

- 先理解，再修改，再验证，再交付
- 优先依据当前任务、`AGENTS.md`、`project-spec.yaml` 和根目录文档工作
- 如果任务输入不够清晰，先补齐任务目标、范围、约束和完成标准
- 在未明确指定前，不默认把 `templates/` 当成当前项目标准
- 优先输出可复用、可迁移、可维护的结果
- 优先减少返工，而不是追求一次生成很多内容

## 2. 任务分级

不同任务需要不同处理方式。

### A 类：可直接执行

满足下面条件时，Agent 可以直接推进：

- 任务目标明确
- 影响范围较小
- 不涉及关键架构分歧
- 不涉及敏感配置、账号、密钥、计费资源或破坏性操作

典型例子：

- 补 README
- 调整脚本入口
- 补 `.env.example`
- 完善文档结构
- 修复局部代码问题

### B 类：建议先说明再执行

满足下面条件时，Agent 应先给出方案摘要，再继续：

- 涉及多个文件或多个模块
- 会引入新的技术选型或依赖
- 会影响开发流程、测试方式或目录结构
- 用户给的是目标，但实现路径不止一种

典型例子：

- 引入数据库迁移工具
- 补前后端分离骨架
- 调整目录结构
- 新增认证模块

### C 类：必须先确认

满足下面条件时，Agent 不应直接做决定：

- 会删除、覆盖、迁移大量现有内容
- 会改变核心架构方向
- 会引入明显成本、部署差异或安全影响
- 需求存在明显冲突
- 存在多个差异很大的合理方案

典型例子：

- 从单体改成前后端分离
- 从 SQLite 改成 PostgreSQL
- 删除历史模板或归档大量文件
- 接入第三方付费服务

## 3. 标准执行顺序

除非任务非常简单，否则默认按这个顺序推进：

1. 读取任务说明和根目录规则
2. 如果任务输入不够结构化，参考 `doc/任务输入模板.md` 先整理任务边界
3. 判断目标、约束、影响范围和风险
4. 如有必要，先向用户确认关键分歧
5. 给出简短执行计划
6. 再进行修改
7. 修改后做最小必要验证
8. 向用户说明结果、变化点、验证方式和剩余风险

## 4. 何时必须先提问

出现下面情况时，应先提问而不是直接动手：

- 用户目标不清楚
- 同时存在两种以上差异很大的实现路径
- 需要决定数据库、认证、部署、前后端拆分等核心方向
- 需要删除、覆盖或迁移已有内容
- 需要用户提供密钥、账号、路径、域名等缺失信息

如果只是存在合理默认值，应先采用默认值并明确说明。

## 5. 文档类任务的交付要求

如果任务主要是文档工作，至少应满足：

- 文档能独立阅读，不依赖聊天上下文
- 说明清楚“先读什么、再做什么、最后验证什么”
- 示例路径和命名都明确标注为可替换
- 对 Agent 的约束写成明确规则，而不是模糊建议
- 必要时给出可直接复用的提示词或命令

## 6. 代码类任务的交付要求

如果任务包含代码或脚本修改，至少应满足：

- 改动与当前任务直接相关，不做无关扩写
- 保持与现有规则、目录和依赖一致
- 尽量通过统一脚本入口运行，而不是依赖个人习惯命令
- 有必要时补最小测试或最小验证步骤
- 修改后说明运行方法、验证方法和限制条件

## 7. 输出内容最低标准

一次合格交付，至少应明确告诉用户这些信息：

- 做了什么
- 为什么这样做
- 修改影响到哪些文件或规则
- 如何验证结果
- 还有哪些未完成项、前提条件或风险

如果任务只完成一部分，也要明确告诉用户当前停在哪一步。

如果仓库中存在 `doc/任务结果输出模板.md`，应优先参考该文档组织最终结果，避免输出结构漂移。

## 8. 完成定义

只有同时满足下面条件，才能算任务完成：

- 目标已被落实到文档、配置、代码或脚本中
- 相关入口文件已同步更新，避免信息分散或互相冲突
- 已做最小必要验证
- 用户可以仅凭仓库内容理解下一步怎么继续

如果只是“写完了内容”，但没有说明怎么用、怎么验证、和什么规则有关，这不算高质量完成。

## 9. 失败与回退原则

如果执行中发现问题，应优先这样处理：

- 先停止继续扩大改动范围
- 明确指出卡点、原因和影响
- 给出可选的下一步方案
- 不在未确认的情况下强行改架构或清理旧内容

如果已有历史内容可能污染上下文，优先做隔离、标注或降权，而不是直接删除。

## 10. 推荐验收清单

在交付前，至少自查一次：

- 当前任务目标是否真的被满足
- 是否引用了正确的规则入口
- 是否误把样例模板当成当前项目标准
- 是否说明了验证方式
- 是否说明了后续怎么接着做

## 11. 一句话记忆

- 先理解
- 再执行
- 再验证
- 最后按规则交付
