# 术语使用规范

这份文档用于统一 `_agent_blueprint` 中高频术语的使用方式。

目标不是把所有词强行改成一个，而是明确：

- 哪个词是主词
- 哪些词只在特定场景使用
- 同一层文档里尽量避免混用近义词

这样可以减少阅读成本，也能降低 Agent 对相近概念的误判。

## 1. 总原则

- 同一概念优先使用同一个主词
- 有语义差异的近义词，保留差异，不强行合并
- 文件名尊重现状，正文表述优先统一
- 先统一入口文档和规范文档，再逐步影响其他资料

## 2. 规则相关术语

### 主词：`规则`

适用：

- 描述 Agent 或协作体系的总行为约束
- 表达最高层、最通用的要求

示例：

- `AGENTS.md` 是总规则入口
- “先读规则，再执行任务”

### 次级词：`准则`

适用：

- 描述执行过程、交付质量、验收方式等更具体的行为要求
- 比“规则”更偏执行标准

示例：

- `任务执行与交付验收准则.md`

### 次级词：`规范`

适用：

- 描述命名、目录、结构、格式等整理类要求
- 比“规则”更偏结构和整理方式

示例：

- `文档命名与目录规范.md`
- `术语使用规范.md`

推荐写法：

- 总体约束优先说“规则”
- 执行标准优先说“准则”
- 结构整理优先说“规范”

## 3. 模板相关术语

### 主词：`模板`

适用：

- 可复用的输入结构、输出结构、项目起步结构
- 示例但带有复用价值的骨架内容

示例：

- `任务输入模板.md`
- `任务结果输出模板.md`
- `project-spec.example.yaml` 作为项目需求模板

### 次级词：`样例`

适用：

- 主要用于说明“这是参考，不是默认标准”
- 多用于提醒不要把示例内容当成唯一答案

示例：

- “`templates/` 是可选样例，不是默认上下文”

### 次级词：`成品模板`

适用：

- 明确指向 `templates/` 里可直接起步的项目模板
- 用于和“规则文档”区分

示例：

- `FastAPI + PostgreSQL + Redis` 成品模板

推荐写法：

- 说复用骨架时优先用“模板”
- 说参考性质时用“样例”
- 说可直接运行的项目骨架时用“成品模板”

## 4. 文档类型术语

### `手册`

适用：

- 分步骤指导完成某项任务
- 更偏操作过程和落地执行

示例：

- `项目初始化操作手册.md`
- `新增接口操作手册.md`

### `教程`

适用：

- 某个主题的教学式说明
- 常用于单点技能配置

示例：

- `GitHub_Gitee_SSH密钥配置一页式教程.md`

### `教材`

适用：

- 更系统、更完整、更偏从零讲解的资料
- 比教程更长、更完整

示例：

- `WSL_Ubuntu_开发环境完整教材.md`

推荐写法：

- 操作流程优先叫“手册”
- 单点教学优先叫“教程”
- 系统完整版优先叫“教材”

## 5. 导航相关术语

### 主词：`导航`

适用：

- 告诉用户“先看什么、再看什么”
- 负责分流和阅读顺序

示例：

- `使用总览导航.md`

### 次级词：`索引`

适用：

- 解释某个目录里有什么、每一层分别放什么
- 适合做目录级入口，而不是阅读顺序分流

示例：

- `doc/README.md`

### 次级词：`总目录`

适用：

- 更偏资料索引和分类集合
- 适合长期知识库入口

示例：

- `开发环境知识库总目录.md`

推荐写法：

- 分流入口优先用“导航”
- 目录入口优先用“索引”
- 资料索引优先用“总目录”

## 6. 当前推荐词表

- 总体行为约束：`规则`
- 执行与验收标准：`准则`
- 命名与结构整理：`规范`
- 可复用结构：`模板`
- 参考性质内容：`样例`
- 可直接起步的项目骨架：`成品模板`
- 分步骤执行文档：`手册`
- 单点教学文档：`教程`
- 系统完整版资料：`教材`
- 阅读分流入口：`导航`
- 目录层级入口：`索引`
- 资料分类索引：`总目录`

## 7. 使用建议

- 新增文档前，先判断自己写的是规则、准则还是规范
- 首页级文档优先使用主词，避免同段混用多个近义词
- 已有文件名如果稳定，不必为了术语统一强行重命名
- 如果正文中出现多个近义词，优先保留一个主词，其余词只在需要强调差异时出现

## 8. 一句话记忆

- 行为看规则
- 执行看准则
- 结构看规范
- 复用看模板
- 步骤看手册
- 分流看导航
- 目录看索引
