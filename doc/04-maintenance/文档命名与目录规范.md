# 文档命名与目录规范

这份文档用于规范 `_agent_blueprint/doc` 下的文档命名和目录组织方式。

目标不是为了形式统一，而是为了减少下面这些问题：

- 新文档越加越多，但别人不知道该放哪里
- 同类文档命名风格混乱，后续很难搜索和维护
- Agent 读取时分不清“规则文档”“操作手册”“知识库资料”“样例模板”
- 分享给别人时，目录结构看起来像临时堆积，而不是一套稳定协议

## 1. 总原则

- 先按用途分组，再决定文件名
- 同一类文档尽量使用同一种命名后缀
- 优先使用能表达用途的中文名称
- 目录结构服务于阅读顺序，优先让新用户知道“现在先看哪一层”
- 只有在分层明显提升可读性和维护性时，才做目录重组

## 2. 当前推荐分组

`doc/` 下的文档，当前按下面四层理解：

### A. `01-onboarding`

这层负责第一次进入仓库时的导航、边界、FAQ 和模板选择。

推荐包含：

- `使用总览导航.md`
- `能力边界与支持范围.md`
- `常见问题与失败处理.md`
- `模板案例总览.md`

### B. `02-workflow`

这层负责任务输入、执行和交付这条协作主链路。

推荐包含：

- `任务输入模板.md`
- `任务执行与交付验收准则.md`
- `任务结果输出模板.md`

### C. `03-reference`

这层负责长期复用的环境、Git、备份和模板资料。

当前代表：

- `开发环境知识库总目录.md`
- `新电脑开发环境一页式速查清单.md`
- `WSL_Ubuntu_开发环境完整教材.md`
- `GitHub_Gitee_SSH密钥配置一页式教程.md`
- `防止重装丢代码与Git远程同步手册.md`
- `Python项目标准模板手册.md`
- `Docker_PostgreSQL_Redis_项目模板手册.md`

命名建议：

- 速查类：使用 `一页式速查清单`
- 详细教程类：使用 `完整教材` 或 `教程`
- 通用操作类：使用 `手册`
- 总索引类：使用 `总目录` 或 `导航`

### D. `04-maintenance`

这层负责维护这套体系本身，而不是指导用户直接起项目。

推荐包含：

- `文档命名与目录规范.md`
- `术语使用规范.md`
- `最小维护说明.md`
- `项目P0迭代路线图.md`

### E. `task-playbooks`

这类文档负责描述某类具体任务怎么做。

当前目录：

- `task-playbooks/`

命名建议：

- 统一使用 `XXX操作手册.md`
- 如果是验证类，也可以使用 `XXX检查手册.md`

示例：

- `新增接口操作手册.md`
- `新增数据表与迁移操作手册.md`
- `改动后验证与回归检查手册.md`

### F. 成品样例与代码模板

这类内容不应放在 `doc/` 下，而应继续放在：

- `templates/`

这样可以避免规则文档和代码样例混在一起。

## 3. 目录使用建议

当前阶段推荐保持下面这个结构：

```text
doc/
├─ 01-onboarding/
│  ├─ 使用总览导航.md
│  ├─ 能力边界与支持范围.md
│  ├─ 常见问题与失败处理.md
│  └─ 模板案例总览.md
├─ 02-workflow/
│  ├─ 任务输入模板.md
│  ├─ 任务执行与交付验收准则.md
│  └─ 任务结果输出模板.md
├─ 03-reference/
│  ├─ 开发环境知识库总目录.md
│  └─ 其他长期知识库资料...
├─ 04-maintenance/
│  ├─ 文档命名与目录规范.md
│  ├─ 术语使用规范.md
│  ├─ 最小维护说明.md
│  └─ 项目P0迭代路线图.md
└─ task-playbooks/
   ├─ README.md
   └─ 各类操作手册
```

这意味着：

- `01-onboarding` 优先放“第一次打开仓库先看什么”
- `02-workflow` 优先放“任务怎么提、怎么做、怎么交付”
- `03-reference` 优先放“长期参考资料”
- `04-maintenance` 优先放“维护者需要看的规则和路线图”
- `task-playbooks/` 专门放“按任务执行”的手册
- `templates/` 专门放“可运行样例和代码模板”

## 4. 新文档应该怎么命名

新增文档前，先判断自己写的是哪一类：

- 如果是告诉别人“先看什么”，优先叫 `导航`
- 如果是解释某个目录里都有什么，优先叫 `索引`，例如目录下的 `README.md`
- 如果是约束 Agent 或团队行为，优先叫 `规则`、`准则`、`规范`
- 如果是给出可复用输入或输出结构，优先叫 `模板`
- 如果是分步骤指导完成某件事，优先叫 `操作手册` 或 `检查手册`
- 如果是长期积累的环境或工具资料，优先叫 `手册`、`教材`、`教程`、`总目录`

## 5. 什么时候需要新建子目录

满足下面条件时，才建议在 `doc/` 下新建子目录：

- 同类文档已经达到 4 到 5 份以上
- 这些文档有明确共同用途
- 把它们单独放在一起能明显降低阅读成本

当前已经固定的子目录有：

- `01-onboarding/`
- `02-workflow/`
- `03-reference/`
- `04-maintenance/`
- `task-playbooks/`

后续如果新增目录，仍要满足“同类文档足够多且能明显降低阅读成本”这两个条件。

## 6. 对现有文档的处理原则

- 已有文档先保持文件名稳定，优先通过目录分层表达用途
- 如果路径已被多个入口引用，迁移时必须同步更新索引、首页和导航
- 只有在命名明显误导、路径明显污染上下文、或已影响后续维护时，才考虑重命名

## 7. 一句话记忆

- 先按用途分层
- 再在层内保持命名统一
- 模板代码留在 `templates/`
- 不为整齐而反复折腾路径
