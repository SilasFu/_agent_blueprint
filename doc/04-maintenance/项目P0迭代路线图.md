# 项目 P0 迭代路线图

这份文档用于保存当前仓库的第一阶段迭代结论，避免关键判断只停留在聊天记录里。

P0 的目标不是把项目包装成“大而全的平台”，而是先把它迭代成一个让 GitHub 新用户能快速理解、顺利跑通、感到靠谱的项目。

## P0 总目标

- 让用户在第一次进入仓库后，知道这是什么
- 让用户知道默认先运行什么命令
- 让用户大概率第一次就能完成初始化
- 让用户知道当前支持什么、不支持什么
- 让用户遇到常见失败时知道下一步怎么做

## 当前判断

- 当前仓库已经具备较强的方法论和规则体系
- 当前仓库已经有脚本入口、模板和文档基础
- 当前仓库更像“有体系的项目启动框架”，还不是“全流程自动化平台”
- 当前最大的缺口不是内容数量，而是主路径体验、能力证明和边界表达

## 目标用户

- 个人开发者
- 1 到 5 人的小团队
- 愿意借助 Agent 提高项目启动效率的人
- 不想从 0 手搓规范、模板、脚本和交付方式的人

## 当前阶段不承诺

- 完整 CI/CD
- 自动部署
- 所有语言和框架全覆盖
- 所有操作系统上的一致自动安装能力
- 从需求到上线完全无人值守

## P0 执行顺序

### 1. 定位收敛

- 用一句话说清仓库定位
- 用一句话说清目标用户
- 用一句话说清当前不做什么

完成标准：

- 新用户看完首页前 30 秒后，能回答仓库是干什么的、适合谁、当前最强能力是什么

### 2. 固化唯一主路径

- 明确 `bootstrap.sh` 是默认主入口
- 明确 `check-env.sh` 是辅助入口
- 明确 `make bootstrap -> make dev -> make test` 是标准顺序

完成标准：

- 文档、脚本、模板、Agent 规则中对默认入口的说法一致

### 3. 打磨主路径体验

- 脚本输出尽量中文
- 缺失工具先说明，再征求安装确认
- 初始化成功后明确告诉用户下一步干什么
- 失败时告诉用户为什么失败、是否还能继续、推荐怎么处理

完成标准：

- 主路径中不出现“只报错、不知道下一步”的状态

### 4. 做出可证明案例

- 至少维护 2 个可复现样例模板
- 给出初始化命令、启动命令、测试命令和访问地址
- 留下足够的仓库内证据，而不是依赖口头解释

完成标准：

- 用户能直接看到“运行什么、得到什么、下一步怎么继续”

### 5. 补齐能力边界与失败场景

- 整理当前支持范围
- 整理推荐环境
- 整理高频失败场景和恢复路径

完成标准：

- 用户不会因为误解能力范围而产生明显落差

当 P0 完成时，这个仓库应该达到下面状态：

- 新用户能快速理解项目定位
- 新用户只需要记住一个默认入口
- 根目录和模板都能用一致方式初始化
