# 项目 P0 迭代路线图

这份文档用于保存当前仓库的第一阶段迭代结论，避免关键判断只停留在聊天记录里。

P0 的目标不是把项目包装成"大而全的平台"，而是先把它迭代成一个让 GitHub 新用户能快速理解、顺利跑通、感到靠谱的项目。

## 当前阶段摘要

- 当前阶段：`P0`（第二轮迭代）
- 当前目标：让新用户快速理解仓库价值、一键创建项目、顺利跑通主路径
- 当前默认主入口：`bash scripts/init-project.sh`（模板创建）或 `make bootstrap`（环境初始化）
- 当前已整理模板：`3` 个
- 当前已补齐的重要能力：
  - `scripts/lib.sh` 共享函数库
  - `scripts/init-project.sh` 模板→项目一键转换
  - `scripts/validate-spec.sh` 需求文件校验（`make validate`）
  - Docker daemon 检测 + 端口冲突检测
  - `compose.yaml` 引用 `.env` 环境变量
  - `project-spec.schema.yaml` 需求文件 schema

## P0 总目标

- 让用户在第一次进入仓库后，知道这是什么
- 让用户知道默认先运行什么命令
- 让用户大概率第一次就能完成初始化
- 让用户知道当前支持什么、不支持什么
- 让用户遇到常见失败时知道下一步怎么做
- 让用户能一键从模板创建项目，不需要手动改多处配置

## 当前判断

- 当前仓库已经具备较强的方法论和规则体系
- 当前仓库已有脚本入口、共享函数库、模板和文档基础
- 当前仓库已有"模板→项目"一键转换能力
- 当前仓库已解决脚本与 Docker 配置的主要健壮性问题
- 当前最大的改进方向是：模板同步自动化、测试覆盖、可视化证明

## 目标用户

- 个人开发者
- 1 到 5 人的小团队
- 愿意借助 Agent 提高项目启动效率的人
- 不想从 0 手搓规范、模板、脚本和交付方式的人

## 当前阶段不承诺

- 完整 CI/CD
- 自动部署
- 所有语言和框架全覆盖
- 所有操作系统上的一致自动安装能力
- 从需求到上线完全无人值守

## P0 执行顺序

### 1. 定位收敛 ✅

- 用一句话说清仓库定位
- 用一句话说清目标用户
- 用一句话说清当前不做什么

### 2. 固化唯一主路径 ✅

- 明确 `init-project.sh` 是模板创建入口
- 明确 `bootstrap.sh` 是环境初始化入口
- 明确 `make bootstrap -> make dev -> make test` 是标准顺序
- 明确 `make validate` 是需求校验入口

### 3. 打磨主路径体验 ✅

- 脚本输出尽量中文
- 缺失工具先说明，再征求安装确认
- Docker daemon 未运行时给出修复建议
- 端口冲突时检测并提示
- 蓝图根目录下引导用户先创建项目
- 初始化成功后明确告诉用户下一步干什么

### 4. 做出可证明案例 ✅

- 维护 3 个可复现样例模板
- 给出初始化命令、启动命令、测试命令和访问地址
- 一键创建命令让用户不需要手动改配置

### 5. 补齐能力边界与失败场景 ✅

- 整理当前支持范围（含 Docker daemon 检测、端口冲突等新能力）
- 整理推荐环境
- 整理高频失败场景和恢复路径

### 6. 脚本健壮性（第二轮新增）✅

- Docker daemon 检测
- 端口冲突检测
- 共享函数库消除重复
- compose.yaml 密码不硬编码
- 需求文件校验

## 完成后的理想状态

当 P0 完成时，这个仓库应该达到下面状态：

- 新用户能快速理解项目定位
- 新用户只需要记住一个创建入口（`init-project.sh`）
- 新用户能一键从模板创建项目，不需要手动改多处配置
- 根目录和模板都能用一致方式初始化
- 至少有 3 个模板可以证明仓库不是"只讲理念"
- 失败场景有明确中文引导
- 文档与脚本表达一致，不互相打架
- Docker 配置安全，密码不硬编码
