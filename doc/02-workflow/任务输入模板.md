# 任务输入模板

这份文档用于约束“用户怎么给 Agent 下任务”。

它解决的问题是：

- 用户描述是否足够清楚
- Agent 是否能快速判断范围和边界
- 是否能减少来回追问
- 是否能让任务更容易高质量完成

如果你还不确定这份文档在整套体系中的位置，可以先看：

- `doc/01-onboarding/使用总览导航.md`

## 1. 总原则

- 先说目标，再说约束，再说范围
- 能明确验收标准，就不要只说“你看着做”
- 能说明不能动什么，就尽量提前说清
- 如果任务只是阶段性目标，要明确说明不是最终完成态

## 2. 最低必含信息

一次高质量任务输入，至少建议包含这些内容：

- 要做什么
- 为什么要做
- 改动范围
- 不希望改什么
- 完成标准
- 是否需要先提方案或先提问

## 3. 短版任务输入模板

适用于你已经知道要改什么，希望快速开始。

```text
任务目标：
<一句话说明这次要做什么>

范围：
<说明允许修改哪些文件、模块或文档>

限制：
<说明不要改什么，或有哪些必须遵守的约束>

完成标准：
<说明做到什么程度算完成>
```

## 4. 长版任务输入模板

适用于影响较大、约束较多或你希望一次说清的任务。

```text
任务背景：
<为什么要做这件事，当前问题是什么>

任务目标：
<本次要达成的结果>

改动范围：
<允许修改哪些目录、文件、模块、文档或脚本>

非目标：
<这次不要顺手扩展哪些内容>

约束条件：
<技术约束、时间约束、风格约束、兼容性要求、保留策略等>

验收标准：
<什么结果算完成，是否要测试、是否要更新文档、是否要可运行>

输出要求：
<希望 Agent 最后如何汇报，例如是否要列修改文件、验证方式、风险和下一步建议>

提问策略：
<是允许 Agent 自主默认，还是遇到关键分歧必须先问>
```

## 5. 常见任务类型示例

### 文档类

```text
任务目标：
补一份规则文档，说明 Agent 在新增接口时的标准执行顺序。

范围：
允许修改 doc 目录和根入口文档。

限制：
不要删除现有模板，不要改业务代码。

完成标准：
文档可独立阅读，并接入 README.md 和 AGENTS.md。
```

### 代码类

```text
任务目标：
给现有后端补一个用户查询接口。

范围：
允许修改后端相关代码、测试和 README。

限制：
不要重构整个项目，不要引入新的数据库方案。

完成标准：
接口可调用，相关参数和返回结构清晰，并补最小验证方式。
```

### 规则类

```text
任务目标：
完善 Agent 的交付规范，让最终结果输出更稳定。

范围：
允许修改 AGENTS.md、README.md 和 doc 目录。

限制：
不要删除已有内容，优先新增规则和入口说明。

完成标准：
新增规则文档，并把它纳入默认读取顺序。
```

## 6. 推荐补充字段

如果任务复杂，建议再补这些信息：

- 当前优先级
- 是否赶时间
- 是否允许加新依赖
- 是否允许改目录结构
- 是否需要兼容旧行为
- 是否需要同步更新测试、文档、脚本

## 7. 不推荐的任务输入方式

下面这些说法容易让 Agent 理解偏差：

- “你帮我弄一下”
- “你看着改”
- “尽量完善一下”
- “顺便把别的也优化了”

更好的做法是明确：

- 本次只做什么
- 这次不要做什么
- 什么时候需要先问我

## 8. 推荐提示词

如果你不想每次自己组织语言，可以直接用这个通用模板：

```text
先阅读 AGENTS.md、project-spec.yaml、doc/02-workflow/任务执行与交付验收准则.md。

任务目标：
<这里写目标>

改动范围：
<这里写允许修改的范围>

限制：
<这里写不能动的内容和必须遵守的约束>

完成标准：
<这里写验收要求>

如果存在关键分歧、明显成本影响或需要删除现有内容，请先向我确认。
最终结果请按 doc/02-workflow/任务结果输出模板.md 的结构汇报。
```
