# 改动后验证与回归检查手册

这份手册用于约束 Agent 在任务完成前必须做的最小验证，避免“改完就结束”。

## 1. 适用场景

- 任何已经修改了文档、配置、脚本或代码的任务
- 任何会影响运行方式、接口行为、数据结构或依赖关系的任务

## 2. 验证目标

验证不是为了追求形式，而是为了回答四个问题：

- 改动是否真的生效
- 是否破坏已有主要路径
- 用户是否知道怎么运行和验证
- 是否还有未覆盖的风险

## 3. 标准执行顺序

1. 确认改动涉及哪些文件和行为
2. 选择最小必要验证方式
3. 能跑诊断就先跑诊断
4. 能跑测试就跑最相关的测试
5. 如果不适合补测试，至少给出手动验证步骤
6. 记录未验证部分和剩余风险

## 4. 最小验证方式

根据任务类型，至少做其中一种或多种：

- 文档任务：检查入口是否一致、说明是否自洽、是否可独立阅读
- 配置任务：检查关键配置项是否补齐、命名是否一致、默认值是否合理
- 脚本任务：检查脚本路径、命令和前置条件是否正确
- 代码任务：运行最相关的测试、诊断或最小启动验证
- 接口任务：说明请求示例、响应示例和错误路径验证方式

## 5. 何时需要补测试

下面场景优先考虑补测试：

- 改动有明确输入输出
- 逻辑存在回归风险
- 项目本身已经有测试体系
- 一个小测试就能显著提升可信度

下面场景可以不强行补测试，但要说明理由：

- 只是文档优化
- 只是入口说明或注释调整
- 当前项目没有测试框架且补测试成本明显高于收益

## 6. 最低交付说明

任务交付时，至少要说明：

- 本次做了哪些改动
- 做了哪些验证
- 哪些没有验证到
- 用户下一步怎么继续

## 7. 常见遗漏

- 只说“已完成”，不说怎么验证
- 改了多个入口文件，但没有检查它们是否一致
- 有明显未验证部分，却没有说明风险
- 把“没时间验证”当成默认状态

## 8. 推荐提示词

```text
先阅读 AGENTS.md、project-spec.yaml、doc/任务执行与交付验收准则.md，
再按 doc/task-playbooks/改动后验证与回归检查手册.md 的要求完成最小验证。
请明确告诉我：你改了什么、验证了什么、还有哪些风险或未验证项。
```
