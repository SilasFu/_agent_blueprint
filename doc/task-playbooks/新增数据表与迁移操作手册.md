# 新增数据表与迁移操作手册

这份手册用于约束 Agent 在涉及数据库结构变更时的执行顺序，避免只改模型不改迁移，或只改迁移不改代码。

## 1. 适用场景

- 新增数据表
- 新增字段、索引、唯一约束
- 调整模型结构
- 新增或修改迁移脚本

## 2. 开始前先确认

先判断下面几点：

- 变更的业务目的是什么
- 是新增实体，还是修改已有实体
- 是否影响已有数据
- 是否需要回填、兼容、默认值或回滚方案
- 当前项目使用什么 ORM 和迁移工具

如果改动涉及历史数据兼容、线上迁移风险或多种建模方式都合理，应先说明再执行。

## 3. 标准执行顺序

1. 阅读规则、项目需求和现有数据层结构
2. 确认表结构、字段类型、约束和关系
3. 更新模型定义或数据库描述文件
4. 生成或编写迁移脚本
5. 同步更新依赖该表的 schema、CRUD、服务或接口
6. 做最小验证，确认迁移后应用能正常工作
7. 记录变更影响和回退注意事项

## 4. 至少应同步检查的内容

- ORM 模型
- 迁移脚本
- 请求和响应 schema
- CRUD 或 repository 层
- 接口层和业务层
- 初始化数据或测试数据
- README 或迁移说明

## 5. 常见遗漏

- 只改模型，不补迁移
- 只建表，不更新读取和写入逻辑
- 改了字段名，但旧代码仍在使用
- 缺少默认值，导致旧数据或新写入失败
- 没说明迁移执行方式和风险

## 6. 最低验收标准

- 模型与迁移一致
- 依赖该表的主要代码路径已同步
- 至少说明了如何执行迁移和如何验证
- 如果存在风险，已明确标注

## 7. 推荐提示词

```text
先阅读 AGENTS.md、project-spec.yaml、doc/任务执行与交付验收准则.md，
再按 doc/task-playbooks/新增数据表与迁移操作手册.md 的顺序完成任务。
请同时处理模型、迁移、schema、CRUD、接口和验证步骤。
如果涉及数据兼容、回填、回滚或线上迁移风险，请先向我说明影响。
```
