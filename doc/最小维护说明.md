# 最小维护说明

这份文档是给以后维护 `_agent_blueprint` 的人看的最短说明。

目标不是解释所有细节，而是避免维护时把入口、规则链路和文档分层改乱。

## 1. 先记住的事

- 先保入口稳定，再补内容
- 先更新规则文档，再更新索引文档
- 先做小步收敛，不要一次性大改全库
- 如果不是必须，不要随便移动或重命名已经被广泛引用的文件

## 2. 这些入口不要随便动

下面这些文件属于稳定入口，修改前要特别小心：

- `README.md`
- `AGENTS.md`
- `项目初始化操作手册.md`
- `doc/使用总览导航.md`
- `doc/开发环境知识库总目录.md`

如果改这些文件，至少要检查它们之间的互相引用是否还成立。

## 3. 推荐维护顺序

当你要新增或调整一条规则时，建议按这个顺序推进：

1. 先改对应规则文档
2. 再改相关模板或手册
3. 再改导航或知识库索引
4. 最后检查首页和总入口

举例：

- 改任务执行标准，先改 `doc/任务执行与交付验收准则.md`
- 改文档结构规则，先改 `doc/文档命名与目录规范.md`
- 改术语定义，先改 `doc/术语使用规范.md`

## 4. 不建议直接做的事

- 不要为了整齐批量重命名现有文件
- 不要把规则文档和 `templates/` 里的代码样例混在一起
- 不要新增一个平行导航，却不更新已有入口
- 不要删除已有内容，除非它已经明显造成冲突、误导或上下文污染

## 5. 新增文档前先判断

新增文档前，先问自己：

- 这是一份规则、准则、规范、模板，还是操作手册
- 它应该放在 `doc/` 根层，还是 `doc/task-playbooks/`
- 它是否已经和现有文档重复
- 新增后是否需要被 `README.md`、`使用总览导航.md` 或知识库索引引用

## 6. 改完后最少检查这些

每次维护后，至少做这几件事：

- 检查最近修改的文档有没有诊断问题
- 检查核心入口之间的互链有没有断
- 检查是否引入了新的重复首页、重复索引或历史副本
- 检查是否把示例路径、个人路径或本机绝对路径写进文档

## 7. 出现冲突时怎么判断

如果不同文档之间出现冲突，优先级建议按下面顺序判断：

1. 当前用户要求
2. `AGENTS.md`
3. 当前任务直接关联的规则文档
4. `README.md` 与 `doc/使用总览导航.md`
5. 知识库资料
6. `templates/` 中的样例内容

## 8. 一句话记忆

- 先改规则
- 再补入口
- 最后做最小检查
