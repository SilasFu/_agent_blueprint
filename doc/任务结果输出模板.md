# 任务结果输出模板

这份文档用于约束 Agent 在任务完成后如何向用户汇报结果。

它解决的问题不是“怎么说得好听”，而是：

- 结果是否完整
- 用户是否能快速看懂
- 用户是否知道怎么验证
- 用户是否知道接下来还能做什么

如果你还不确定这份文档在整套体系中的位置，可以先看：

- `doc/使用总览导航.md`

## 1. 总原则

- 先给结果，再给细节
- 先说已完成内容，再说验证和风险
- 不要只说“搞定了”或“已完成”
- 输出要帮助用户继续推进，而不是让用户再次追问

## 2. 最低必含信息

一次合格的任务结果输出，至少应包含这些内容：

- 做了什么
- 改了哪些文件或规则
- 如何验证
- 还有哪些未覆盖项、限制或风险
- 如果合适，给出下一步建议

## 3. 推荐输出结构

推荐按下面顺序组织最终结果：

### A. 已完成

先用 1 到 3 条说明本次完成了什么，聚焦结果，不要重复执行过程。

### B. 修改位置

说明改动主要落在哪些文件、哪些规则、哪些入口。

### C. 验证情况

说明做了哪些诊断、测试、手动验证，或为什么暂时没有补测试。

### D. 风险与边界

说明哪些内容没有覆盖、有哪些前提条件、是否还有残留风险。

### E. 下一步建议

如果存在自然衔接的后续动作，可以给出 1 到 3 条建议。

## 4. 文档类任务的推荐话术

适用于规则整理、说明文档、操作手册、目录结构说明等任务。

```text
已完成：
- 已新增/更新相关文档，并把入口接入到主规则链路中。
- 已明确读取顺序、使用方式和关键边界，减少后续误读。

修改位置：
- <文件 A>
- <文件 B>

验证情况：
- 已检查文档结构和入口一致性。
- 已完成相关诊断，当前无新增问题。

风险与边界：
- 当前只完成了文档与规则层收敛，尚未扩展到具体业务实现。

下一步建议：
- 可以继续补 <某类规则/手册/模板>，进一步提高 Agent 的执行稳定性。
```

## 5. 代码类任务的推荐话术

适用于代码、脚本、配置或接口变更。

```text
已完成：
- 已实现 <目标功能>，并同步处理相关入口与依赖项。

修改位置：
- <文件 A>
- <文件 B>
- <文件 C>

验证情况：
- 已运行 <诊断/测试/命令>。
- 已确认 <关键路径> 可用。

风险与边界：
- <未覆盖项或前提条件>。

下一步建议：
- 如有需要，可以继续补 <测试/文档/扩展能力>。
```

## 6. 不推荐的输出方式

下面这些输出方式应尽量避免：

- 只说“好了”
- 只列文件名，不说结果和影响
- 只说改了什么，不说怎么验证
- 不说明未验证项和边界
- 一上来长篇讲过程，用户却不知道最后改成了什么

## 7. 何时需要更详细说明

下面场景建议适当增加说明：

- 改动涉及多个模块
- 变更影响运行方式或开发流程
- 有未完成项或风险需要用户决策
- 本次只是阶段性交付，不是全部完成
